# A to Z Store - Grocery & Services Delivery Application

## Overview

A to Z Store is a mobile-first grocery delivery and home services web application built with React and Express. Contact: 9999878381 Users can browse fresh vegetables, fruits, and dairy products, add items to cart, and place orders with scheduled delivery slots. The app includes user authentication with an admin approval workflow, a referral system, and an admin dashboard for managing customers and orders.

## User Preferences

Preferred communication style: Simple, everyday language.

## System Architecture

### Frontend Architecture
- **Framework**: React 18 with TypeScript
- **Routing**: Wouter (lightweight React router)
- **State Management**: Zustand for global client state (user auth, cart, products)
- **Data Fetching**: TanStack React Query for server state management
- **UI Components**: shadcn/ui component library with Radix UI primitives
- **Styling**: Tailwind CSS v4 with custom theme variables
- **Animations**: Framer Motion for product card animations
- **Forms**: React Hook Form with Zod validation

### Backend Architecture
- **Runtime**: Node.js with Express.js
- **Language**: TypeScript with ESM modules
- **Build Tool**: esbuild for server bundling, Vite for client
- **Development**: tsx for TypeScript execution without compilation

### Data Storage
- **ORM**: Drizzle ORM configured for PostgreSQL
- **Schema Location**: `shared/schema.ts` contains database table definitions
- **Current Storage**: In-memory storage implementation (`MemStorage` class) as placeholder
- **Database Ready**: PostgreSQL configuration present via `DATABASE_URL` environment variable

### Authentication Flow
- Client-side auth state managed in Zustand store
- User approval workflow: new signups require admin approval before accessing the app
- Admin users have access to customer management dashboard
- Protected routes redirect unauthenticated users to `/auth`

### Key Design Patterns
- **Storage Interface**: `IStorage` interface abstracts data operations, allowing swap between in-memory and database implementations
- **Path Aliases**: `@/` maps to client/src, `@shared/` maps to shared folder
- **API Pattern**: All backend routes should be prefixed with `/api`
- **Component Library**: shadcn/ui components in `client/src/components/ui/`

### Referral Wallet System
- **Referral Bonus**: ₹50 credited to referrer's wallet when referred user's order status changes to "Delivered" (idempotent - only credits once per order)
- **Checkout Integration**: Users with referral balance see a toggle switch in CartDrawer to apply credit; deducted from total on order placement
- **Schema**: `referralBalance` (integer) on users table, `referralDiscount` (integer) on orders table
- **API**: `GET /api/user/:id/referral-balance` returns wallet balance; `POST /api/orders` accepts `applyReferralCredit` flag
- **Admin View**: Order cards and detail modal show referral discount line; PDF invoice includes referral credit
- **Referrals Page**: Shows wallet balance card when balance > 0
- **Auto-Apply**: Referral credit is automatically applied at checkout (no toggle required); server checks balance inside DB transaction
- **Atomic Transaction**: Order creation + item insertion + referral balance deduction happen in a single DB transaction to prevent race conditions
- **Admin Display**: Shows Gross Amount / Referral Discount / Net Payable in order detail modal and PDF invoice; customer cards show wallet balance badge
- **Startup Fix**: Server assigns referralCode to any user missing one on startup (prevents orphaned referrals)
- **Admin Credit API**: `POST /api/admin/users/:id/credit-referral` with `{amount, adminId}` for manual wallet crediting (capped at ₹10,000)

### P&L Dashboard (Profit & Loss / Accounts)
- **Page**: `/admin/accounts` — AdminAccounts.tsx with Dashboard, Transactions, Add tabs
- **Schema**: `transactions` table (id, type, amount, description, itemName, quantity, purchasePrice, billImage, date, createdAt)
- **Transaction Types**: Purchase (Mandi/Supplier), Offline Sale, Expense
- **Bill Image Upload**: Uses object storage presigned URLs for Mandi bill photos
- **P&L Summary API**: `GET /api/admin/pnl-summary` — calculates totals from delivered orders (Online Sales) + transactions
- **Date Filters**: All Time, Today, This Week, This Month, Custom range
- **Type Filters**: All, Purchase, Offline Sale, Expense
- **Quick Link**: Purple "P&L" button on Admin.tsx quick links bar
- **Staff Tracking**: Transactions store `staffId` and `staffName` to identify who entered data; admin P&L view shows "by StaffName" badge

### Staff System (Purchase & Sale Manager)
- **Role**: `isStaff` and `staffStatus` fields on users table
- **Auth Pages**: `/staff-auth` (StaffAuth.tsx) for login/signup, `/staff-pending` (StaffPending.tsx) for approval wait
- **Dashboard**: `/staff` (StaffDashboard.tsx) — only Purchase and Offline Sale entry forms, no access to profits/customers
- **Approval Flow**: Staff signup creates account with `staffStatus: "pending"`; admin approves via Staff tab in Admin panel
- **Admin Management**: "Staff" tab in Admin.tsx with approve/deactivate/delete/view entries per staff
- **API Routes**: `POST /api/staff/register`, `POST /api/staff/login`, `GET /api/admin/staff`, `POST /api/admin/staff/:id/approve|reject`, `GET /api/staff/transactions?staffId=`, `DELETE /api/admin/staff/:id`
- **Security**: Staff cannot see Net Profit, Total Revenue, customer data, or referral settings

### Service Booking System
- **ServiceCard Component**: `client/src/components/product/ServiceCard.tsx` — professional card with "Book Now" button + "Call Request" button, booking form modal, success confirmation
- **Booking Form**: Bottom sheet with name, phone, address, preferred date/time, special instructions
- **Call Request Form**: Green bottom sheet with name, phone, and reason for call — submits as `requestType: "call"` to same API
- **Schema**: `service_requests` table (id, serviceId, serviceName, servicePrice, customerName, customerPhone, customerAddress, preferredDate, preferredTime, notes, requestType, status, createdAt)
- **Request Types**: `booking` (default, for service bookings) and `call` (for call requests where customer wants to discuss before ordering)
- **Statuses**: Pending → Confirmed → Completed (or Cancelled)
- **API Routes**: `GET /api/service-requests`, `POST /api/service-requests`, `PATCH /api/service-requests/:id/status`, `DELETE /api/service-requests/:id`
- **Admin Tab**: "Requests" tab in Admin.tsx — view all requests, confirm/cancel/complete, call/WhatsApp customer, delete; call requests show green "CALL REQUEST" badge and highlighted reason
- **Admin Add Request**: Admin can manually create service requests from Requests tab via "Add Request" button
- **Notification**: Blue badge (📞) in admin header for pending requests; separate green badge for pending call requests
- **Integration**: ServiceCard used in Categories.tsx, Home.tsx (both grid and horizontal scroll for service categories)

### Seller System (Multi-Vendor)
- **Schema**: `isSeller`, `shopName`, `sellerStatus`, `allowedCategories` (text[]) on users table
- **Auth Pages**: `/seller-auth` (SellerAuth.tsx) for login/signup, `/seller-pending` (SellerPending.tsx) for approval wait
- **Dashboard**: `/seller` (SellerDashboard.tsx) — sellers can add/edit/delete their own products
- **Allowed Categories**: Admin selects which product categories a seller can use when approving or via "Manage Categories" button
- **Backend Enforcement**: Both `POST /api/seller/products` and `PUT /api/seller/products/:id` validate category against seller's `allowedCategories`
- **Admin Management**: "Sellers" tab in Admin.tsx with approve (with category picker modal), reject, delete, and manage categories for approved sellers
- **API Routes**: `POST /api/seller/register`, `POST /api/seller/login`, `GET /api/seller/products`, `POST /api/seller/products`, `PUT /api/seller/products/:id`, `DELETE /api/seller/products/:id`, `POST /api/admin/sellers/:id/approve`, `POST /api/admin/sellers/:id/reject`, `POST /api/admin/sellers/:id/categories`

### Customer Auto-Approval
- **No approval required**: New customers are automatically approved on registration (`approvalStatus: "approved"`)
- **Direct access**: Users can sign up and start shopping immediately
- **Admin notification**: Green badge shows new customers (last 24 hours) with address details in Customers tab
